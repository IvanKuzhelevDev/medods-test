# МИС — Трекер рабочих задач (API)

API модуля трекера рабочих задач для персонала медицинской информационной системы (МИС).
Врачи и администраторы ставят себе рабочие задачи (провести операцию, обзвонить пациентов,
сделать обход, подготовить отчёт), в том числе **повторяющиеся**, и ведут их по дням.

Тестовое задание для junior+ Ruby on Rails разработчика.

## Стек

- **Ruby 3.4**, **Rails 8.1** (API-only)
- **PostgreSQL 16**
- **RSpec** + FactoryBot — тесты
- **rswag** — Swagger/OpenAPI документация
- **Pagy** — пагинация
- **Docker / docker-compose** — запуск одной командой

## Быстрый запуск

Нужен только Docker (Docker Desktop или совместимый рантайм).

```bash
docker compose up --build
```

- API:      <http://localhost:3000>
- Swagger:  <http://localhost:3000/api-docs>

При старте контейнер сам создаёт и мигрирует БД и засевает обязательные системные теги.
БД PostgreSQL поднимается в контейнере; на хост проброшен порт `5433` (чтобы не конфликтовать
с локальным PostgreSQL).

### Тесты

```bash
docker compose run --rm web bash -c "bin/rails db:prepare && bundle exec rspec"
```

> Примечание про старый Docker: команда `docker compose build` требует buildx ≥ 0.17.
> Если его нет, соберите образ классическим билдером и поднимите стек обычным `up`:
> ```bash
> DOCKER_BUILDKIT=0 docker build -t test-medods-web .
> docker compose up
> ```

## Архитектура повторяемости

Это ключевая часть задания. Реализован подход **«правило + виртуальные вхождения + материализация исключений»** (как в календарях):

- **`Task`** — определение/серия задачи: название, описание, статус, теги и **правило повторения**.
  Разовая задача — это серия с типом `once`.
- **`TaskOccurrence`** — конкретный экземпляр на дату. **Создаётся в БД только когда отклоняется
  от правила** (отметили выполненным, перенесли, отменили). Пока не тронут — он «виртуальный»,
  вычисляется на лету.
- **`Recurrence::Generator`** (`app/services/recurrence/`) — чистый движок: по задаче и окну
  `[from, to]` возвращает конкретные даты. Поддерживает типы: `once`, `daily` (каждый n-й день),
  `monthly` (числа 1..31), `specific_dates`, `parity` (чётные/нечётные дни).
- **`Occurrences::Calendar`** — разворачивает правила в окне и накладывает материализованные
  исключения, выдавая плоский список вхождений.

Как закрыты «критичные» моменты из ТЗ:

| Проблема | Решение |
|---|---|
| **Бесконечность** (серия без даты окончания) | Вхождения никогда не создаются пачкой — только генерируются под запрошенное окно. Есть дефолтное окно (текущий месяц) и жёсткий потолок `MAX_WINDOW_DAYS = 366`. |
| **Независимое состояние экземпляра** | Эффективный статус дня = статус материализованного `TaskOccurrence`, иначе дефолт серии. Отметка «выполнено» за сегодня создаёт строку только за этот день — соседние не затрагиваются. |
| **Исключения из правил** (бонус) | Перенос пишет `scheduled_at`, отмена — `canceled=true`; якорная дата `occurrence_date` сохраняется, поэтому правило не «регенерит» этот день. |

## Обзор API (`/api/v1`)

Приём — разделение **серий** (определения) и **календаря** (вхождения):

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/tasks` | список задач (фильтры `status`, `tag`) |
| POST | `/tasks` | создать задачу (разовую или повторяющуюся) |
| GET | `/tasks/:id` | показать задачу |
| PATCH | `/tasks/:id` | изменить всю серию |
| DELETE | `/tasks/:id` | удалить задачу |
| GET | `/occurrences` | **календарь**: вхождения за период (`from`, `to`, `status`, `tag`) |
| PATCH | `/tasks/:task_id/occurrences/:date` | изменить один день (статус/перенос/правка) |
| DELETE | `/tasks/:task_id/occurrences/:date` | отменить один день |
| GET | `/tags` | список тегов |
| POST | `/tags` | создать тег |
| PATCH | `/tags/:id` | переименовать (системные → 422) |
| DELETE | `/tags/:id` | удалить (системные → 422) |
| POST | `/tasks/:task_id/tags` | добавить тег задаче |
| DELETE | `/tasks/:task_id/tags/:id` | снять тег с задачи |

Вхождение адресуется парой `(task_id, date)`, где `date` — `YYYY-MM-DD`.

### Примеры (curl)

Создать ежедневную задачу:

```bash
curl -X POST http://localhost:3000/api/v1/tasks \
  -H 'Content-Type: application/json' \
  -d '{"task":{"title":"Обзвон пациентов","recurrence_type":"daily","recurrence_interval":1,"starts_on":"2026-06-01","due_time":"10:00"}}'
```

Календарь за период:

```bash
curl 'http://localhost:3000/api/v1/occurrences?from=2026-06-01&to=2026-06-07'
```

Отметить конкретный день выполненным (материализует исключение только за этот день):

```bash
curl -X PATCH http://localhost:3000/api/v1/tasks/1/occurrences/2026-06-03 \
  -H 'Content-Type: application/json' \
  -d '{"occurrence":{"status":"done"}}'
```

Отменить один день («пропустить сегодня»):

```bash
curl -X DELETE http://localhost:3000/api/v1/tasks/1/occurrences/2026-06-04
```

## Принятые допущения

ТЗ намеренно не детализировано; ниже — разумные решения, принятые по ходу:

- **Аутентификация и пользователи не реализованы** — в ТЗ это не требуется, остаёмся в рамках
  предметной области трекера.
- **Статусы**: `new`, `in_progress`, `done`, `canceled`. Переходы свободные (без жёсткого
  state machine — отмечено как точка расширения). Хранятся строкой (не Rails enum, чтобы
  значение `new` не конфликтовало с `ActiveRecord#new`).
- **Даты** — календарные, в одной таймзоне (по умолчанию UTC); время выполнения в течение дня
  хранится отдельно в `due_time`.
- **Вхождение** адресуется парой `(task_id, date)`; перенос (`scheduled_at`) сохраняет якорную
  дату слота `occurrence_date`.
- **Бессрочные серии** требуют ограниченного окна запроса: дефолт — текущий месяц, потолок —
  `MAX_WINDOW_DAYS = 366`.
- **Системные теги** `отчётность`, `операции`, `звонок` создаются сидами и защищены от изменения
  и удаления на уровне модели (`throw :abort`), что отражается как `422` в API.
- Материализованное вхождение можно создать только на дату, реально входящую в расписание серии.

## Структура проекта

```
app/
  models/            task, tag, task_tag, task_occurrence
  services/
    recurrence/      generator — движок дат повторения
    occurrences/     calendar (проекция окна) + view (value object)
  serializers/       plain-ruby сериализаторы
  controllers/api/v1 base, tasks, occurrences, tags, task_tags
spec/
  models/ services/ requests/    тесты
  integration/                   rswag-спеки (источник swagger.yaml + реальные тесты)
swagger/v1/swagger.yaml          сгенерированная OpenAPI-спека
```

## Что внутри

- Чистое разделение «серия задач ↔ одиночный экземпляр» в API и доменной модели.
- Тонкие контроллеры, бизнес-логика в сервис-объектах, единый JSON-формат ошибок.
- Тесты: модели, сервисы (движок повторяемости покрыт подробно), request- и rswag-спеки.
